# Финальная приёмка системных улучшений

Дата фиксации: 2026-07-18.

Этот документ связывает требования issue #28 с реализованными пакетами и контрольными проверками. Он используется как регрессионный чек-лист при последующих изменениях дашборда.

## Пакет 1 — фильтры и состояния

Реализация: PR #33, #34.

- множественный выбор географии и категорий;
- алфавитная сортировка категорий;
- поиск в длинных фильтрах;
- корректное отображение фокусного объекта с городом;
- корректные состояния all/custom и сообщения о фокусном объекте;
- мобильная панель фильтров без переполнения и iOS zoom.

Контроль: TypeScript, unit, Playwright desktop/mobile, accessibility.

## Пакет 2 — данные «Скоро открытие»

Реализация: PR #35, #36.

- исключение брендов, уже работающих в том же объекте;
- нормализация брендов, регистра, алиасов и статусов;
- очистка выполняется в pipeline, а не в UI;
- production использует каноническую команду построения агрегатов.

Контроль: data-quality и legacy tests.

## Пакет 3 — таблицы и визуализация

Реализация: PR #37, #38, #39, #40.

- управляемый скролл потенциальных брендов;
- sticky headers, стабильные колонки и mobile-card layout;
- доступный popover «Ещё»;
- фильтры и поиск таблицы «Скоро открытие»;
- обновлённая heatmap и контраст интерфейса;
- Chromium/WebKit browser QA;
- устранено расхождение production и e2e-сборки.

Контроль: TypeScript, unit, browser QA, Lighthouse, production-equivalent aggregates.

## Пакет 4 — методология и терминология

Реализация: PR #41.

- определения эксклюзивов и пересечений;
- описание индекса Жаккара и ограничений метода;
- пояснение формирования потенциальных брендов;
- единая терминология desktop/mobile;
- сокращены нечитаемые служебные подписи.

Контроль: frontend-quality, browser QA, Lighthouse, legacy tests.

## Публикация и открытые вкладки

Реализация: PR #42.

- открытая вкладка проверяет `build-info.json`;
- во время deployment переходит на maintenance-экран;
- после публикации автоматически загружает новый build с cache-busting параметром;
- логика ограничена GitHub Pages и не влияет на локальную разработку.

## Обязательная проверка следующих изменений

Перед merge изменения, затрагивающего UI, данные или workflow:

1. Проверить TypeScript, unit и production build.
2. Перестроить production-equivalent aggregates.
3. Выполнить browser QA в Chromium и WebKit, включая mobile.
4. Проверить accessibility и отсутствие горизонтального overflow.
5. Проверить Lighthouse.
6. После merge подтвердить успешные jobs `quality` и `deploy`; rollback не должен запускаться.
7. Проверить `build-info.json`: `status=production`, `app=tenant-mix-react`, `build` соответствует merge commit.
