# Tenant Mix Analytics

Профессиональное React-приложение для сравнительного анализа tenant mix торговых центров России. Production-данные не подменяются демонстрационными значениями; все KPI, benchmark, таблицы и экспорты используют единый текущий срез.

Опубликованная версия: [GitHub Pages](https://koptelov-vk.github.io/tenant-mix-dashboard/).

## Статус React-миграции

React-версия реализована в ветке `agent/react-dashboard-migration` и публикуется только после успешного quality job и ручного merge draft PR. Предыдущий Dashboard 2.0 сохранён в `legacy/dashboard-v2-pre-react.html`; он не удаляется при миграции.

Реализовано:

- Vite, React 18.3, TypeScript strict, Zustand, TanStack Query и TanStack Virtual;
- единый `AnalysisContext` для KPI, медиан, графиков, таблиц, уникальности, Жаккара, gap-анализа и экспорта;
- страницы «Обзор», «Сопоставимость», «Категории», «Бренды», «Сценарии», «Скоро открытие», «Качество данных» и empty state истории;
- URL-state фокуса, раздела, группы, категории, городов, ТЦ, качества источника, `gapN`, GLA/GBA и метрики;
- CSV и lazy XLSX текущего среза;
- mobile layout 320/390 px без горизонтального overflow;
- Zod-валидация production JSON, Vitest, Playwright и axe smoke;
- quality-gated GitHub Pages deployment.

## Расчётная модель

### Единый срез

`focusMall`, `peerMalls` и `displayMalls` разделены. Если фокусный объект не соответствует критериям группы, он остаётся на экране, но не включается в медиану peer group. Категория, бренд, источник и качество применяются до расчёта зависимых показателей.

### Уникальность

- **Во всей базе**: нормализованный бренд встречается ровно в одном ТЦ среди всех объектов production-базы.
- **В выбранной группе**: бренд встречается ровно в одном объекте текущей области сравнения.
- **Эксклюзив фокусного ТЦ**: бренд есть в фокусном ТЦ и отсутствует у выбранных конкурентов.
- **Пересечение**: бренд фокусного ТЦ представлен хотя бы у одного выбранного конкурента.

Область расчёта показывается рядом с показателем.

### Плотность

```text
Плотность брендов = число различных брендов / подтверждённая GLA × 10 000
Плотность категории = число брендов категории / подтверждённая GLA × 10 000
```

GBA никогда не подставляется вместо GLA. При отсутствии подтверждённой GLA результат равен `н/д`, а объект исключается из медианы плотности.

### Жаккар

```text
J(A, B) = |A ∩ B| / |A ∪ B|
```

Индекс отражает сходство состава нормализованных брендов, но не доказывает полную коммерческую сопоставимость объектов.

### Медиана и категорийные отклонения

Медиана рассчитывается только по текущей peer group и тому же фильтрованному срезу, что KPI. Для категории доступны количество, доля и плотность; разные метрики не смешиваются.

### Потенциальные бренды

Бренд попадает в список, если отсутствует в фокусном ТЦ, присутствует минимум у `N` выбранных конкурентов, соответствует категории и имеет источник требуемого качества.

Сигнал отражает распространённость бренда и качество данных, но не готовность к сделке. Он не учитывает стратегию, доступную площадь, технические требования и коммерческие условия.

## История и сценарии

Production-набор содержит один сопоставимый снимок. Поэтому временные графики и KPI trends не формируются; приложение показывает объясняющий empty state.

Сценарии сохраняются в `localStorage` и не изменяют baseline. Они пересчитывают только доступные tenant mix показатели. NOI, ставки, выручка, трафик, площадь магазина и коммерческий эффект не рассчитываются без данных.

## Deployment

Workflow `.github/workflows/pages.yml` выполняет:

1. валидацию исходных данных;
2. пересборку агрегатов;
3. legacy contract tests;
4. React unit и Zod production tests;
5. production build;
6. Playwright desktop/mobile/axe;
7. три Lighthouse-прогона;
8. загрузку QA-артефактов;
9. deploy `dist` только из `main` и только после успешного `quality`.

При падении обязательного шага новая версия не публикуется, поэтому предыдущая GitHub Pages deployment остаётся доступной.

## Rollback

1. Выбрать последний успешный GitHub Pages deployment или revert merge-коммита React-миграции.
2. Запустить workflow из восстановленного `main`.
3. Для срочного статического fallback использовать `legacy/dashboard-v2-pre-react.html` как `index.html` отдельным коммитом после проверки.

## Сценарное моделирование

Сценарий хранит только добавленные и исключённые нормализованные бренды и не изменяет production-базу. Для baseline и scenario единым расчётом пересчитываются количество брендов, категории и их доли, эксклюзивы фокусного объекта, пересечения и индекс Жаккара по каждому объекту peer group.

Сценарии можно сохранять, загружать, переименовывать, копировать, удалять, импортировать и экспортировать в JSON. Импорт проходит проверку схемы версии 1; дата исходного снимка сохраняется, а при несовпадении с текущим снимком интерфейс показывает предупреждение. NOI, ставки, выручка, трафик, площадь магазина и коммерческий эффект без исходных данных не рассчитываются.

Не выполнять ручную замену production-данных и не подменять GLA значением GBA.

## Ограничения

- База отражает снимок официальных и открытых источников, а не юридический реестр договоров.
- Источники различаются по качеству и дате проверки.
- Отсутствие бренда в каталоге не доказывает отсутствие действующего договора.
- Историческая динамика требует минимум двух сопоставимых снимков.
- Сценарии и gap-анализ не являются коммерческой рекомендацией.
- Локальный Lighthouse на Windows может завершаться ошибкой очистки профиля Chrome; обязательный quality gate выполняется в Ubuntu CI.
