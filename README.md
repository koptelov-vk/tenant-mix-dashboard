# Tenant Mix Dashboard 2.0

Управленческий аналитический продукт для директора по аренде, коммерческого директора, управляющего ТЦ и asset manager. Dashboard 2.0 добавлен совместимым слоем поверх подтвержденной базы: исходные строки, ссылки, категории и нормализованные ключи не подменяются браузерными расчетами.

Опубликованная версия `main`: [GitHub Pages](https://koptelov-vk.github.io/tenant-mix-dashboard/).

Версия 2.0 разрабатывается в ветке `agent/dashboard-v2-ux` и публикуется только после успешного CI и отдельного слияния pull request.

## Информационная архитектура

- **Обзор**: фокусный ТЦ, управленческие KPI, ключевые выводы, профиль tenant mix, медиана, сходство и потенциальные бренды для рассмотрения.
- **Сравнение ТЦ**: GLA/GBA, город, класс, бренды, плотность, глобальная уникальность, структура категорий и Жаккар.
- **Tenant mix**: heatmap, медиана, три вида уникальности, пересечения и gap-анализ.
- **Арендаторы**: полнотекстовый поиск, фильтры, сортировка, виртуализированный реестр, мобильные карточки и CSV.
- **Скоро открытие**: объект, бренд, категория, даты, источник и уровень подтверждения.

Фантастика является фокусным объектом по умолчанию. Фокус, вкладка, группа, география, категория, выбранные ТЦ, метрика, порог gap-анализа и расширенные фильтры сохраняются в URL. Back, Forward и Reload восстанавливают состояние.

## Показатели и формулы

### Бренды фокусного ТЦ

Количество различных нормализованных брендов фокусного объекта в текущем срезе.

### Уникальность

- **Уникальный во всей базе**: нормализованный бренд присутствует ровно в одном ТЦ среди всех 30 объектов.
- **Уникальный в выбранной группе**: бренд присутствует ровно в одном объекте текущей группы сравнения.
- **Эксклюзив фокусного объекта**: бренд есть в фокусном ТЦ и отсутствует у всех выбранных конкурентов.

Область расчета всегда отображается в интерфейсе. Изменение фильтра не изменяет смысл показателя скрытно.

Доля эксклюзивов:

```text
Эксклюзивы фокусного объекта / бренды фокусного объекта × 100%
```

### Пересечение с группой

Количество брендов фокусного объекта, которые встречаются хотя бы у одного выбранного конкурента. Доля рассчитывается от состава фокусного объекта.

### Плотность по GLA

```text
Плотность брендов = количество брендов / подтвержденная GLA × 10 000
Плотность категории = бренды категории / подтвержденная GLA × 10 000
```

Единица измерения: брендов на 10 000 м² GLA. Если подтвержденной GLA нет, результат равен `нет данных`. GBA никогда не используется вместо GLA.

### Индекс сходства

Используется коэффициент Жаккара по нормализованным брендам:

```text
J(A, B) = |A ∩ B| / |A ∪ B|
```

Результат показывается в процентах вместе с числом общих брендов. Это сходство состава брендов, а не доказательство полной коммерческой сопоставимости объектов.

### Сравнение с медианой

Для каждой категории рассчитываются значение фокусного ТЦ, медиана, минимум, максимум, позиция и отклонение от медианы внутри текущей группы. Доли категории рассчитываются от числа брендов конкретного ТЦ.

### Потенциальные бренды для рассмотрения

В список попадает бренд, который:

1. отсутствует в фокусном ТЦ;
2. присутствует минимум у `N` выбранных конкурентов;
3. соответствует выбранной категории;
4. имеет источник в базе.

Порог `N` меняется пользователем и сохраняется в URL. Присутствие у конкурентов не означает готовность бренда к сделке и не учитывает стратегию, площадь, технические требования и коммерческие условия.

### Executive Summary

Формируются только воспроизводимые утверждения: место по числу брендов, отклонения категорий от медианы, доля эксклюзивов, число gap-брендов и наиболее близкий объект по Жаккару. Причинные и коммерческие рекомендации автоматически не генерируются.

## Данные и агрегаты

Основной источник: `data/processed/База_финальная.csv`.

При сборке создаются:

```text
data/aggregates/
  mall_summary.json
  category_matrix.json
  brand_presence.json
  mall_similarity.json
  brand_gaps.json
  data_quality_summary.json
  dashboard_data.json
```

Некорректные HTML-заглушки `javascript:void(0)` не считаются URL. Их исходные значения сохранены в `data/processed/invalid_source_url_audit.csv`, а поле ссылки в аналитическом наборе оставлено пустым.

## Структура проекта

```text
src/
  index.html
  styles/
  components/
  app.js
data/
  processed/
  aggregates/
scripts/
tests/
legacy/

```

`src/` является редактируемым источником. `index.html` собирается как автономный файл для совместимости и GitHub Pages. Предыдущая версия сохранена в `legacy/dashboard-v1.html`.

## Локальная сборка

```powershell
python scripts/validate_data.py
python scripts/build_aggregates.py
python scripts/build_dashboard_v2.py
pnpm test
pnpm test:e2e
```

Поиск в реестре использует debounce 200 мс. Длинная desktop-таблица виртуализирована; вкладки рендерятся по требованию.

## Проверки

CI выполняет:

1. установку Python и Node.js;
2. валидацию данных;
3. пересчет агрегатов;
4. сборку автономного HTML;
5. unit tests;
6. Playwright desktop и mobile;
7. проверку URL-state, CSV и пустого результата;
8. accessibility smoke по WCAG A/AA;
9. проверку JavaScript errors и переполнения на 390 px;
10. загрузку QA-артефакта;
11. deploy только для успешного push в `main`.

Job публикации зависит от `quality`; при любой обязательной ошибке деплой не запускается.

Локальный результат текущей ветки: 6 unit-тестов и 17 Playwright-сценариев пройдены, 1 desktop-дубль mobile-only проверки пропущен по условию.

Проверяются обязательные поля, пустые бренды и ключи, пара `ТЦ + нормализованный бренд`, категории, URL, положительные GLA/GBA, условие `GLA ≤ GBA`, города, статусы источников, формулы уникальности и сокращение каталогов более чем на 10% относительно baseline.

## Известные ограничения

- База отражает снимок открытых источников, а не юридический реестр договоров.
- Восемь объектов не имеют подтвержденной GLA; плотность для них намеренно не рассчитывается.
- История открытий и закрытий требует нескольких сопоставимых временных снимков. Текущий файл не позволяет достоверно восстановить прошлые состояния.
- Автоматический gap-анализ не учитывает коммерческие условия и не является рекомендацией заключить сделку.
- Расширенные фильтры брендов и арендаторов используют браузерный мультивыбор; сохраненные пользовательские представления остаются задачей P1.
- Data Quality Dashboard как отдельная вкладка остается задачей P1; текущая сводка качества формируется при сборке и блокирует публикацию при критических ошибках.
